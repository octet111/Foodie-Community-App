# フーディコミュニティ運営アプリ 要件定義書 v1.1

作成日: 2026-06-08 / 更新日: 2026-06-28 / ステータス: 確定（v1.1: 例外系・受け入れ条件・ステータス遷移を追加。§9 に決定ログ。**フェーズ13 追補（2026-06-28）を §17 に追記**）

## 1. 目的とスコープ

招待制フーディコミュニティの「企画・募集・精算・体験共有」を一元化し、運営負荷と企画ハードルを下げる。希少な食体験という価値の源泉（排他性）は維持し、参加の偏りだけを解消する。

本要件定義のゴールは、あったら良い機能を含む全体像の定義と、技術選定・アーキテクチャまでを含む。開発体制は一人 + AI駆動開発を前提とし、実装容易性・運用負荷の低さを優先軸とする。

スコープ外（企画書準拠 + 本要件での確定）:
- アプリ内決済・送金（資金移動業規制回避。集金は計算・管理のみ）
- 招待制ロジックの実装（文化としては維持するが、MVPでは機能化しない）
- LINEミニアプリ化（将来検討）
- 完全オープン化

## 2. 前提・確定事項（ヒアリング結果）

| 論点 | 確定内容 |
|---|---|
| 認証 | メール+パスワードを基本。Supabase Auth を採用 |
| 集金 | 割り勘計算 + 未払いチェック管理まで。送金はしない（銀行振込は手動） |
| 割り勘粒度 | 参加パート単位（一次会・二次会等）で金額が変わる方式 |
| 割り勘の母集団 | 参加表明をベースに自動生成。幹事と指定立替者が最終調整可能 |
| 未払い可視性 | 幹事・立替者に加え、参加者本人も自分の支払状況を閲覧可能 |
| リマインド | まずはアプリ内通知 + メール。LINE投稿連携は将来 |
| AI紹介文生成 | MVPでは入れない（背骨完成後に追加）。ただし店リンクのOGP取得は最初から |
| ストックリスト | MVPに含める（行きたい店ストック） |
| 利用デバイス | スマホ・PC両方を同等重視（レスポンシブ必須） |
| 技術基盤 | Supabase（Auth / Storage / Postgres / Edge Functions / pg_cron）。Webアプリ |
| メール送信基盤 | Resend を採用。Supabase Edge Functions から送信。テンプレートは React Email |
| 表示名と認証 | ニックネームのみ公開表示。メールは認証用で非公開（他メンバーから見えない） |
| リマインド回数 | **4日前 + 当日**の2回を自動送信（アプリ内通知 + メール）。時刻は管理者が設定画面で変更可 |
| 想定規模 | メンバー20〜50名・月間企画2〜5件。単一コミュニティでの利用を前提とした小規模設計 |
| 希少性機能 | 予約難易度タグ＋希少バッジ、「確保できます」宣言をMVPに含める |

## 3. ペルソナ・ロール

- **幹事/古参層**: 企画作成、募集管理、集金管理を行う。運営負荷の軽減が主便益。
- **新規/準新規メンバー**: 企画を眺めて参加。低ハードルでの参加・将来的な企画デビューが主便益。

ロール（権限）:
| ロール | 主な権限 |
|---|---|
| メンバー | 企画閲覧・参加表明・自分の支払状況確認・店ストック・写真投稿 |
| 企画者（幹事） | 自企画の作成・編集・締切・集金管理・母集団調整 |
| 立替者 | 企画者から指定され、割り勘母集団・金額の最終調整が可能 |
| 管理者 | メンバー管理（将来の招待管理の受け皿） |

### 3.1 権限管理（認可）

**基本方針**: 「誰が・どのデータに・何をできるか」を二層で守る。
1. アプリ層（UI）: 権限のないボタン・操作を出さない（UX）
2. DB層（Supabase RLS）: Postgres の Row Level Security でDBレベルから拒否する（最後の砦）

一人運用ではフロントの作り込みミスが起きうるため、RLSによるDB層の防御を必須とする。アプリ層のチェックだけに依存しない。

**ロールの考え方**:
- グローバルロール（`users.role`）: member / admin の2種。コミュニティ全体に対する権限。
- リソース単位ロール: 「企画者」「立替者」はグローバルではなく企画ごとに決まる。`events.organizer_id` と、立替者指定（`settlements.finalized_by` または別途指定テーブル）で判定する。

**CRUD権限マトリクス**（C=作成 / R=閲覧 / U=更新 / D=削除。括弧はスコープ）:

| リソース | 一般メンバー | 企画者（自企画） | 立替者（自企画） | 管理者 |
|---|---|---|---|---|
| events（企画） | C, R | C, R, U, D※1 | R | C, R, U, D※1 |
| participations（参加表明） | C, R, D（自分・締切前のみ） | ＋R(全員), D（締切後の取消含む） | 自分の分のみ（一般と同等） | 全件 C, R, U, D |
| settlements / settlement_items（精算・明細） | R（自分の明細のみ） | C, R, U※2 | R, U※2 | C, R, U, D |
| comments（コメント） | C, R ／ U, D（自分の投稿のみ） | 一般と同等 | 一般と同等 | C, R, D（U不可）※3 |
| photos（写真・フェーズ2） | C, R ／ D（自分の投稿のみ） | ＋D（自企画の全写真） | 一般と同等 | C, R, D |
| stocks（店ストック） | C, R, U, D（自分の分のみ） | ― | ― | C, R, U, D |
| shops（店） | C, R | U（`area`/`rarity`、登録者のみ） | ― | C, R, U, D |
| secure_claims（確保宣言） | C, R ／ U, D（自分の宣言のみ） | ― | ― | C, R, U, D |
| community_settings（コミュニティ設定） | R | ― | ― | R, U |

※1 企画のDは論理削除（`deleted_at`）。Uには募集締切・定員変更を含む
※2 Uには母集団調整・請求額調整・支払済チェック・確定/確定取消を含む
※3 管理者は不適切コメントの削除のみ可、内容の編集（U）は不可
※4 店（shops）のDは登録者または管理者。有効企画（`events.deleted_at is null`）がある場合は店データは削除不可（ストックのみ削除）

**確定した重要ルール**:
- 企画は企画者本人と管理者以外は編集・削除できない（RLSで `auth.uid() = organizer_id OR role = admin` を強制）。
- 参加表明・取消は本人のみ操作可（admin は参加者管理可）。**店ストック**は本人のみ操作可（**admin は全件削除可**）。締切後の取消は企画者（幹事）・管理者のみ。
- **店（shops）**の `area`/`rarity` 編集は登録者のみ。削除は登録者または admin（有効企画がある場合は店データは残る）。
- 支払状況は本人は自分の分のみ、企画者・立替者は自企画の全員分を閲覧可。
- 企画へのコメントは投稿者本人のみ編集・削除できる（RLSで `auth.uid() = user_id` を強制）。例外として管理者は不適切投稿への対応のため削除のみ可、編集は不可。

**確定した重要ルール**:
- 企画は企画者本人と管理者以外は編集・削除できない（RLSで `auth.uid() = organizer_id OR role = admin` を強制）。
- 参加表明・取消は本人のみ操作可（admin は参加者管理可）。店ストックは本人のみ操作可（admin は全件削除可）。
- 店（shops）の `area`/`rarity` 編集は登録者のみ。削除は登録者または admin（有効企画がある場合は店データは残る）。
- 支払状況は本人は自分の分のみ、企画者・立替者は自企画の全員分を閲覧可。

## 4. 機能要件

### 4.1 MVP（背骨）— 必須

各機能に受け入れ条件（AC: これを満たせば完成）を付す。AI駆動開発時はACをそのままテスト仕様として使う。

1. **認証** — メール+パスワード登録・ログイン・ログアウト・パスワードリセット（Supabase Auth）
   - AC: 未ログインでは企画閲覧・操作不可／パスワードリセットメールが届きリセット完了できる
2. **企画作成**
   - 店リンク貼付 → OGP（og:image / og:title / og:description）を自動取得しカード表示。対応データソースは食べログ・Google Maps・その他主要グルメサイト（一休、Retty、ぐるなび等）を含む汎用OGP取得
   - OGP取得失敗時（画像なし・リンク切れ・メタ情報不足）は、店名の手動入力＋画像の手動アップロードで作成を継続できる
   - 日時（**10分単位**。開催日 + 時刻プルダウン）・場所・定員・概要（企画説明: プレーンテキスト・改行のみ）・参加パート（一次会/二次会/三次会…可変）を設定
   - ※スクレイピングは行わず、公開メタ情報（OGP）取得に限定
   - AC: 主要グルメサイト・Google MapsのURL貼付でカードが自動生成される／OGPが取れないURLでも手動入力で企画を公開できる
3. **募集・参加表明**
   - 参加ボタンで表明、パートごとの参加可否を選択
   - 定員到達で自動締切（パート単位の定員管理可）
   - キャンセルは募集中（締切前）のみ本人が操作可。締切後は幹事へ連絡し、企画者（幹事）が手動で外す
   - 定員到達後にキャンセルが出た場合、空き分は再度募集中に戻る（自動繰り上げ・ウェイトリストはフェーズ2）
   - AC: capacity到達と同時に該当パートのstatusがclosedになり参加ボタンが無効化される／締切後は本人のキャンセルボタンが非表示になり、企画者のみ参加者を外せる
4. **企画一覧・カレンダー** — 全メンバーに「いつ・どこ・誰」を可視化
   - AC: 募集中・開催予定の企画が一覧とカレンダーの両形式で表示され、各企画の参加者（ニックネーム）が見える
5. **自動リマインド** — 開催**4日前**と当日の2回、参加者へアプリ内通知 + メール送信（pg_cron + Edge Function + Resend）
   - メール送信失敗時はリトライせず、管理者へ失敗を通知する（小規模運用のため自動復旧は持たない）
   - 送信時刻・メール件名/本文テンプレートは管理者が設定画面で変更可
   - AC: **4日前・当日**に参加者全員へ通知とメールが届く／送信失敗が発生した場合、管理者に失敗内容が通知される
6. **集金管理（管理のみ・送金なし）**
   - パートごとの参加実績から割り勘額を自動計算（端数は円単位で切り上げ）
   - 母集団は参加表明ベースで自動生成、企画者（幹事）・立替者が最終調整可
   - 精算確定後は明細をロック。「確定取消」操作でロックを解除し再編集できる
   - 未払い/支払済チェック管理。参加者本人も自分の状況を閲覧可
   - 集金連絡（金額・振込先などのテキスト連絡）
   - AC: 一次会5,000円・二次会3,000円で二次会不参加者の請求が5,000円になる／確定後は金額編集UIが無効化され、確定取消で再度編集できる

   **実装反映（フェーズ6 完成・2026-06-16）** — 画面 S09 `/events/[id]/settlement`

   | 項目 | 実装内容 |
   |---|---|
   | 割り勘計算 | `ceil(パート実額 ÷ 参加人数)` をパートごとに算出し合算（`src/lib/settlement.ts`） |
   | 実費デフォルト | 精算画面初回表示時、未保存の実費は **想定費用 × 参加人数**（`settlements.part_actuals` jsonb に保存） |
   | 編集の保存方式 | 実費・パートチェック・請求額手入力・行追加/除外は **下書きプレビュー →「変更を保存」で一括反映**。支払チェックのみ即時DB保存 |
   | 母集団調整 | 精算画面でパート参加のチェック変更・メンバー追加/除外が可能（保存で `participations` / `settlement_items` を同期） |
   | 手動請求額 | 幹事が請求額を上書き可能（`settlement_items.adjusted_by` に記録）。プレビュー中は手動上書きと自動計算を分離管理 |
   | 表示 | 幹事画面は集金合計のみ表示（実額・差分はDB保持、画面非表示）。参加者は本人の請求額・支払状況のみ閲覧 |
   | 確定 | 未保存変更がある間は確定不可。確定後はDBトリガで明細ロック |
   | 権限 | 幹事・立替者は全明細、参加者は本人明細のみ（RLS + RPC `get_my_settlement_for_event`） |
   | テスト | ユニットテスト + E2E（実費→保存→支払→確定→編集拒否） |

7. **店ストックリスト** — 行きたい店をストック。ストックから企画作成へ導線
   - AC: ストックした店から1タップで企画作成画面に遷移し、店情報が引き継がれる
8. **希少性の可視化** — 店の希少性属性＋コネの「確保できます」宣言
   - 店に予約難易度タグを設定可（ふらっと入れる / 予約可 / 紹介制 / 数ヶ月待ち / 会員制）。企画カード・店ページに希少バッジとして表示
   - メンバーは「この店、確保できます」を宣言でき、店とコネ保有者が紐づく。宣言時にコネ種別（常連 / 店主と知人 / 紹介ルート保有 / 会員 / その他）と補足条件を設定可。確保宣言から企画作成への導線を持つ
   - AC: 紹介制タグの店の企画カードに希少バッジが表示される／確保宣言した店の一覧に宣言者のニックネームとコネ種別が表示され、宣言から企画作成画面に遷移できる
9. **コミュニティ設定** — コミュニティ名・ロゴ・振込先・リマインド時刻デフォルト・リマインドメールテンプレートの設定
   - 管理者がコミュニティ名とロゴ画像（Supabase Storage）を設定でき、ヘッダー等の全画面共通部分に表示される
   - 設定は下部タブ/サイドナビの「設定」（admin のみ）からアクセス。マイページ内の設定リンクは持たない
   - AC: 管理者が設定画面で名前・ロゴを変更すると全メンバーの画面に反映される／一般メンバーには設定変更UIが表示されない

### 4.2 例外系・異常系ルール（確定）

| # | ケース | 決定ルール |
|---|---|---|
| 1 | 締切後のキャンセル | 本人操作は不可。幹事へ連絡し企画者（幹事）が手動で参加を外す |
| 2 | 定員到達後のキャンセル発生 | 自動繰り上げなし。空きが出たら募集中に戻るのみ（ウェイトリストはフェーズ2） |
| 3 | 参加表明つき企画の削除 | 削除可だが論理削除（`deleted_at` を立てて非表示。参加・精算データは保持） |
| 4 | 精算確定後の修正 | 確定後はロック。「確定取消」操作で再編集可（取消操作は企画者・立替者・管理者のみ） |
| 5 | OGP取得失敗 | 店名の手動入力＋画像の手動アップロードで企画作成を継続できる。対応ソースは食べログ・Google Maps・主要グルメサイト全般の汎用OGP取得 |
| 6 | リマインド送信失敗 | リトライなし。管理者へ失敗通知のみ |

### 4.3 フェーズ2 — あったら良い（背骨完成後）

- AI企画紹介文生成（店情報をもとに魅力紹介文を生成）
- 開催後アルバム（写真投稿・Supabase Storage）・店情報の蓄積
- ウェイトリスト（定員到達後の参加待ち登録・キャンセル時の自動繰り上げ＋通知）

- 古参のお墨付きコメント（希少性の可視化）
- 過去企画アーカイブ・振り返り・統計ダッシュボード（初期指標: ①開催件数・参加人数の推移〈月次・年次〉 ②企画者ランキング〈誰が何件企画したか〉 ③行った店マップ・ジャンル分布）
- 領収書発行補助
- メンバー紹介・招待管理

### 4.4 将来検討

- LINE連携（通知・企画投稿）
- 汎用化（他コミュニティ展開）
- フリーミアム有料機能。課金の軸足は「効率化機能（高度AI生成・一括管理・自動化テンプレート等）」と「高度化機能（BI/分析ダッシュボード・行った店マップ・統計）」を主軸とする。希少情報の限定公開は無料側の価値の源泉として維持し、課金対象とはしない
- 招待管理の機能化（方向性は未確定。招待制・ニックネーム文化の維持を前提に、将来検討としてメモのみ）
- 統計ビジュアル化機能（フェーズ2の統計ダッシュボードを参照）

| ② | 参加表明・定員管理が手作業 | 参加ボタン＋定員自動締切（§4.1-3） | MVP |
| ③ | 「いつ・どこ・誰」が管理者しか分からない | 企画一覧・カレンダー（§4.1-4） | MVP |
| ④ | 企画ストックの保存場所がない | 行きたい店ストックリスト（§4.1-7） | MVP |

| ⑥ | 投稿・リマインド・集金連絡が手作業 | 自動リマインド（§4.1-5）＋集金管理（§4.1-6） | MVP |

| ⑧ | 企画者が固定化 | 誰でも投稿可（権限設計§3.1で担保）＋企画数可視化・トロフィー（§4.3） | MVP（投稿可）＋フェーズ2（可視化） |
| ⑨ | 参加者が偏る | **機能化しない**。低ハードル企画・日程バリエーションの運用ルールで対応 | 運用 |
| ⑩ | 希少性・コネが活きない | 店の希少性属性（予約難易度タグ・希少バッジ）＋「確保できます」宣言（§4.1-8）／古参のお墨付きコメント（§4.3） | **MVP**（属性・宣言）＋フェーズ2（お墨付き） |

## 5. 中核フロー

店ストック or 「確保できます」宣言から企画作成（OGPカード化・希少バッジ）→ 募集・パート別参加表明・定員自動締切 → 自動リマインド（通知+メール）→ 開催 → 集金管理（パート別割り勘計算・母集団調整・未払いチェック・連絡）→（フェーズ2）写真・店情報をアーカイブ化

ポイント:
- 割り勘は `event_parts`（パート）単位で金額が変動 → `settlement_items.amount`（請求額）はパート参加実績の合算で算出
- 端数処理は切り上げ（各人の負担額は円単位で切り上げる）。切り上げにより集金合計が実額を上回る差分（おつり）は幹事/立替者が預かる扱いとし、`settlements` に集金合計・実額・差分を保持する
- 母集団は `participations`（参加表明）から自動生成し、`settlements` 確定時に幹事・立替者が `settlement_items` を上書き調整可能
- 1企画につき精算は必ず1つ（`events 1 --- 1 settlements`）。一次会・二次会・三次会以上のパートは `event_parts` で可変に保持し、何パート増えても精算は1つに集約する
- RLS（Row Level Security）で「本人は自分の支払状況のみ／幹事・立替者は自企画全体」を制御
- コメントは投稿者本人のみ編集・削除可（RLSで `auth.uid() = user_id` を強制。管理者は不適切投稿対応のため削除可）

## 7. 非機能要件

- **プラットフォーム**: Webアプリ、スマホ/PC両対応のレスポンシブ
- **認証/認可**: Supabase Auth + Postgres RLS
- **通知**: pg_cron によるスケジュール + Edge Function での送信（Resend経由）。**4日前 + 当日**の2回
- **可用性/運用**: 一人運用前提。マネージドサービス（Supabase）中心で運用負荷最小化
- **規模・キャパシティ**: メンバー20〜50名・月間企画2〜5件を前提とした小規模設計。メール送信は最大でも月500通程度（50名×2回×5件）でResend無料枠（月3,000通）内。DBは年間数百レコード規模のため、特別なインデックス設計・スケーリング対策・キャッシュ層は持たない（YAGNI）。Supabase・Vercelとも無料枠での運用を基本とする
- **法令**: 決済・送金を持たず資金移動業規制を回避。OGP取得は公開メタ情報に限定し各サイト規約を尊重
